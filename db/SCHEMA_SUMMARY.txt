================================================================================
NUST University Database - Schema Summary & Design Decisions
Scope: Undergraduate (Bachelor's) Programs Only
DBMS: MySQL 8.0+ (InnoDB)
================================================================================

OVERVIEW
--------
The database models two modules that share the `program` and `term` tables:

  1. ACADEMIC MODULE
     school, faculty, program, course, prerequisite, program_course,
     student, term, classroom, section, enrollment

  2. ADMISSIONS MODULE
     applicant, entry_test, test_attempt, application, offer

The bridge between modules: when an applicant accepts an offer, a new
`student` row is created with an optional FK `applicant_id` pointing back
to the originating applicant, and `program_id` taken from the accepted
application.

KEY DESIGN DECISIONS
--------------------
1. All primary keys are manually-assigned VARCHAR(20) alphanumeric IDs
   (no AUTO_INCREMENT); this mirrors NUST's real identifiers
   (e.g. 'SEECS', 'STU-001', 'CS351').
2. `course.course_code` is the PK of course (per the prompt exception).
3. Grades are stored as an ENUM on `enrollment.grade`; GPA points are
   mapped in queries via a CASE expression (no lookup table).
4. `enrollment.grade` is NULLABLE: NULL means "in progress, not yet graded".
5. `enrollment.attendance_percentage` is tracked; a BEFORE UPDATE trigger
   (`trg_enrollment_xf`) assigns grade 'XF' automatically when attendance
   falls below 75.00.
6. `applicant.best_test_score` is a denormalized best-score cache,
   maintained by an AFTER INSERT trigger on `test_attempt`
   (`trg_best_test_score`).
7. `application` snapshots `high_school_score` and `best_test_score` at
   submission time (so later test attempts do not retroactively change
   existing applications). `aggregate_score = snapshot_hs + snapshot_best_test`.
8. No separate merit-list tables; merit ranking is a RANK() query over
   `application`.
9. Semester tracking is handled by three columns:
     program.total_semesters   (8 for BS/BE/BBA, 10 for B.Arch/LLB)
     student.current_semester  (1..10, starts at 1)
     program_course.recommended_semester (advisory)
   A cross-check query counts distinct terms a student has enrolled in.
10. Course type drives contact hours: Theory = credit*1, Lab/Project = credit*3.
    `contact_hours` is physically stored and kept consistent by the INSERT data.
11. ON DELETE RESTRICT is the default for FKs; CASCADE is used only for
    junction tables (prerequisite, program_course, test_attempt, enrollment).
12. Indexes exist on every FK column; composite UNIQUE on
    application(applicant_id, program_id, term_id) prevents duplicate
    applications; composite PK on prerequisite(course_code, prereq_course_code)
    prevents duplicate prereq edges.

TRIGGERS
--------
- trg_enrollment_xf       BEFORE UPDATE on enrollment   (attendance < 75 => XF)
- trg_best_test_score     AFTER  INSERT on test_attempt (refresh applicant.best_test_score)

GRADING SCALE
-------------
A=4.0  B+=3.5  B=3.0  C+=2.5  C=2.0  D+=1.5  D=1.0
F=0.0  XF=0.0  I=incomplete  W=withdrawn
(I and W excluded from GPA calculations; XF counts as 0.0 fail.)

================================================================================
EXACT CARDINALITIES BETWEEN ENTITIES
================================================================================

ACADEMIC MODULE
---------------
school       1 ---- 1..*  faculty             (school must employ >=1 faculty; faculty.school_id NOT NULL)
school       1 ---- 1..*  program             (school exists to offer >=1 program; program.school_id NOT NULL)
school       1 ---- 1..*  course              (school owns >=1 course; course.school_id NOT NULL)
program      1 ---- 0..*  student             (student.program_id NOT NULL)

course       *  ---- *    course              (self-ref via prerequisite; 0..* on both sides)
program      *  ---- *    course              (via program_course; each side 1..*; junction
                                               adds recommended_semester)

course       1 ---- 0..*  section             (section.course_code NOT NULL)
term         1 ---- 0..*  section             (section.term_id NOT NULL)
classroom    1 ---- 0..*  section             (section.classroom_id NOT NULL)
faculty      1 ---- 0..*  section             (section.faculty_id NOT NULL)

student      *  ---- *    section             (via enrollment; junction carries grade
                                               and attendance_percentage)

ADMISSIONS MODULE
-----------------
applicant    1 ---- 1..*  application         (an applicant submits >=1 application; application.applicant_id NOT NULL)
program      1 ---- 0..*  application         (application.program_id NOT NULL)
term         1 ---- 0..*  application         (application.term_id NOT NULL; always Fall)
applicant    *  ---- *    entry_test          (via test_attempt; an applicant takes >=1 NET, so
                                               applicant 1..* test_attempt; junction carries score)
application  1 ---- 0..1  offer               (offer.application_id UNIQUE; offer optional)

CROSS-MODULE LINK
-----------------
applicant    0..1 -- 0..1 student             (student.applicant_id NULLABLE;
                                               populated when an offer is Accepted;
                                               optional on both sides)

================================================================================
CARDINALITY JUSTIFICATIONS (real-world reasoning, per relationship)
================================================================================

Notation: 1 = exactly one (mandatory), 0..1 = zero-or-one (optional),
          1..* = one-or-many (mandatory-many), 0..* = zero-or-many (optional-many).

ACADEMIC MODULE
---------------
school       1  ---- 1..*  faculty
    A school cannot operate without faculty; at least one faculty member is
    required to deliver any program. Faculty.school_id is NOT NULL, so a
    faculty member always belongs to exactly one school.

school       1  ---- 1..*  program
    A school exists precisely to offer programs; a school with zero programs
    is not meaningful in NUST's structure. Program.school_id is NOT NULL.

school       1  ---- 1..*  course
    Courses are owned by the school that delivers them, and a functioning
    school always owns at least one course in its catalogue.
    Course.school_id is NOT NULL.

program      1  ---- 0..*  student
    A newly launched program may have zero admitted students until its first
    intake closes. Once admissions run, students accumulate over time.
    Student.program_id is NOT NULL, so every student has exactly one program.

program      1  ---- 0..*  application
    A freshly created program may have zero applications before its first
    admission cycle opens. Application.program_id is NOT NULL.

program      1..* ---- 1..*  course              (via program_course)
    Every program must require at least one course (else it is not a
    curriculum), and every course in the catalogue must be mapped to at
    least one program (otherwise it would not be offered to anyone).

course       0..* ---- 0..*  course              (self-reference via prerequisite)
    Foundational courses have no prerequisites, and advanced courses may
    themselves not be prerequisites of anything else; both sides are
    therefore optional-many.

course       1  ---- 0..*  section
    A course exists in the catalogue year-round but is not necessarily
    offered every term; some terms yield zero sections for a given course.
    Section.course_code is NOT NULL.

term         1  ---- 0..*  section
    A newly created future term holds zero sections until the scheduling
    committee publishes them. Section.term_id is NOT NULL.

term         1  ---- 0..*  application
    A future admission term has zero applications until the intake opens.
    Application.term_id is NOT NULL.

classroom    1  ---- 0..*  section
    A classroom may sit unused in a given term (renovation, low demand,
    reserved capacity), so zero sections per classroom is valid.
    Section.classroom_id is NOT NULL.

faculty      1  ---- 0..*  section
    A faculty member on sabbatical, administrative duty, or research leave
    may teach zero sections in a given term. Section.faculty_id is NOT NULL.

student      0..* ---- 0..*  section              (via enrollment)
    A newly admitted student in the pre-registration window has zero
    enrollments; a section in the same window likewise has zero registrants
    until add/drop closes. Both sides are therefore optional-many.

ADMISSIONS MODULE
-----------------
applicant    1  ---- 1..*  application
    Being an applicant is defined by having submitted at least one
    application; a row in `applicant` with zero applications would not
    exist in the first place. Application.applicant_id is NOT NULL.

applicant    1  ---- 1..*  test_attempt
    NUST admission requires sitting the NET at least once; an applicant
    with zero test attempts cannot be evaluated. TestAttempt uses a
    composite PK (applicant_id + test_id) with applicant_id NOT NULL.

entry_test   1  ---- 0..*  test_attempt
    A scheduled but not-yet-held test has zero attempts recorded.
    TestAttempt.test_id is NOT NULL.

application  1  ---- 0..1  offer
    Only a subset of applications result in an offer (others are rejected,
    withdrawn, or still pending); each application yields at most one
    offer (offer.application_id is UNIQUE).

CROSS-MODULE LINK
-----------------
applicant    0..1 ---- 0..1 student
    Optional on both sides: an applicant may never convert (rejected or
    declined => no student row), and the schema allows student.applicant_id
    to be NULL as a data-migration escape hatch for legacy/imported student
    records whose originating applicant is not tracked.

================================================================================
TABLE INVENTORY
================================================================================
  1. school            (PK school_id)
  2. faculty           (PK faculty_id;  FK school_id)
  3. program           (PK program_id;  FK school_id)
  4. course            (PK course_code; FK school_id)
  5. prerequisite      (PK course_code + prereq_course_code)
  6. program_course    (PK program_id + course_code;  recommended_semester)
  7. term              (PK term_id;  UQ term_name + academic_year)
  8. classroom         (PK classroom_id)
  9. student           (PK student_id; FK program_id, FK applicant_id NULL)
 10. section           (PK section_id; FK course_code, term_id, classroom_id, faculty_id)
 11. enrollment        (PK student_id + section_id; attendance, grade)
 12. applicant         (PK applicant_id)
 13. entry_test        (PK test_id)
 14. test_attempt      (PK applicant_id + test_id; score)
 15. application       (PK application_id; UQ applicant_id+program_id+term_id)
 16. offer             (PK offer_id; UQ application_id)
