================================================================================
NUST University Database - Schema Summary & Design Decisions
Scope: Undergraduate (Bachelor's) programs only (BS, BE, BBA, BArch)
DBMS:  MySQL 8.0+ (InnoDB, utf8mb4)
================================================================================

OVERVIEW
--------
The database models two tightly connected pipelines that meet at the
applicant-becomes-student boundary:

  1. ADMISSIONS PIPELINE
     applicant, entry_test, test_attempt, application, offer

  2. ACADEMICS PIPELINE
     school, faculty, program, course, prerequisite, program_course,
     term, classroom, section, enrollment, student

A selected `application` leads to an `offer`; the student accepts by the
insertion of a `student` row, which fires a trigger that flips the matching
application to 'Enrolled' and the offer to 'Accepted'. The financial ledger
(fees, challans) is intentionally NOT modelled in this schema — it was
removed from the design.

KEY DESIGN DECISIONS
--------------------
1. VARCHAR primary keys throughout. Entity IDs are human-readable codes,
   not surrogate auto-increment integers:
       program_id = program code   (e.g. 'BSCS', 'BESE', 'BBA')
       course_code is itself the PK of `course` (e.g. 'CS220')
       school_id  = abbreviation   (e.g. 'SEECS', 'NBS')
       term_id    = compact code   (e.g. 'FA25' for Fall 2025)
   This makes joins and seed data readable without surrogate lookup tables.

2. Student carries DIRECT FKs to both `program` (program_id) and `applicant`
   (applicant_id) — a deliberate denormalization over reaching these through
   `application`. UNIQUE(applicant_id) on `student` enforces the 1:1
   applicant-becomes-student cardinality. Trade-off: shorter joins at query
   time vs a BCNF violation on `student` (see
   Phase2_Normalization_Analysis.md §2).

3. Course is owned by a school (course.school_id), NOT by a program.
   program <-> course is a true M:N relationship captured in the
   `program_course` junction, which carries recommended_semester and is_core
   on the junction row itself.

4. Prerequisites are modelled as a self-referential junction on `course`
   (table: `prerequisite`), keyed by (course_code, prereq_course_code). A
   course can have many prereqs and can be the prereq of many other courses.
   A CHECK constraint prevents a course from being its own prerequisite.

5. `offer` is separated from `application`. It has its own issue_date,
   expiry_date, and a status lifecycle ('Issued' -> 'Accepted' / 'Declined'
   / 'Expired'). UNIQUE(application_id) enforces at most one offer per
   application.

6. `enrollment` is keyed by the composite (student_id, section_id). Grade
   is an ENUM that is NULL while the section is still in progress or a
   grade has not yet been reported (no separate status column).
   attendance_percentage is a live metric and may be NULL.

7. ON DELETE CASCADE on the cleanup paths: dropping a program, applicant,
   test, course, or application sweeps its junction/child rows.
   Other FKs use default RESTRICT to protect academic records.

8. Indexes exist on high-traffic FK and filter columns (see NUST.sql
   `INDEXES` block): enrollment.section_id, section.*,
   application.status/program_id/term_id, test_attempt.test_id, offer.status,
   student.program_id, prerequisite.prereq_course_code, faculty.school_id.

9. The schema ships with two views (student_transcript,
   classroom_utilization), two stored procedures (accept_admission,
   transfer_student_to_section), and one function
   (is_eligible_for_engineering).

TRIGGERS
--------
- enforce_class_capacity            BEFORE INSERT on enrollment
      Rejects the row if the hosting section's classroom is at capacity.

- auto_update_application_status    AFTER INSERT on student
      Matches by (applicant_id, program_id). Promotes the 'Selected'
      application to 'Enrolled' and flips the linked offer from 'Issued'
      to 'Accepted' in the same pass.

GRADE LETTERS
-------------
Allowed values on enrollment.grade:
    'A', 'A-', 'B+', 'B', 'B-', 'C+', 'C', 'C-', 'D+', 'D', 'F'
grade NULL means the enrollment is in progress or no grade has been posted.

================================================================================
EXACT CARDINALITIES BETWEEN ENTITIES
================================================================================

ACADEMIC STRUCTURE
------------------
school       1 ---- 1..*  program          (program.school_id NOT NULL;
                                            a program belongs to exactly one school)
school       1 ---- 0..*  faculty          (faculty.school_id NOT NULL)
school       1 ---- 0..*  course           (course.school_id NOT NULL;
                                            courses are owned by their teaching school)

program      *  ---- *    course           (via program_course; junction adds
                                            recommended_semester + is_core)

course       *  ---- *    course           (via prerequisite; self-referential
                                            M:N, course cannot be its own prereq)

course       1 ---- 0..*  section          (section.course_code NOT NULL;
                                            a course need not have a section
                                            every term)
term         1 ---- 0..*  section          (section.term_id NOT NULL)
faculty      1 ---- 0..*  section          (section.faculty_id NOT NULL;
                                            every section has exactly one instructor)
classroom    1 ---- 0..*  section          (section.classroom_id NOT NULL)

student      1 ---- 0..*  enrollment       (enrollment.student_id NOT NULL)
section      1 ---- 0..*  enrollment       (enrollment.section_id NOT NULL)
                                            PK is composite (student_id, section_id)

ADMISSIONS PIPELINE
-------------------
applicant    1 ---- 0..*  application      (application.applicant_id NOT NULL)
program      1 ---- 0..*  application      (application.program_id NOT NULL)
term         1 ---- 0..*  application      (application.term_id NOT NULL)
                                            UNIQUE (applicant_id, program_id, term_id):
                                            at most one application per
                                            (applicant, program, term) cell

applicant    *  ---- *    entry_test       (via test_attempt; UNIQUE
                                            (applicant_id, test_id) — an
                                            applicant sits each test at most once)

application  1 ---- 0..1  offer            (UNIQUE offer.application_id;
                                            at most one offer per application)

CROSS-PIPELINE LINK
-------------------
applicant    1 ---- 0..1  student          (UNIQUE student.applicant_id;
                                            only an admitted applicant becomes a student)
program      1 ---- 0..*  student          (student.program_id NOT NULL —
                                            direct FK, deliberately denormalized
                                            from application)

================================================================================
TABLE INVENTORY (16 tables)
================================================================================
  1. school          (PK school_id;        UQ school_name, abbreviation)
  2. faculty         (PK faculty_id;       FK school_id; UQ email)
  3. program         (PK program_id;       FK school_id;
                                           UQ (school_id, program_name))
  4. course          (PK course_code;      FK school_id)
  5. prerequisite    (PK (course_code, prereq_course_code);
                      FK both cols -> course)
  6. program_course  (PK (program_id, course_code);
                      FK program, course)
  7. term            (PK term_id;          UQ (term_name, academic_year))
  8. classroom       (PK classroom_id;     UQ (building, room_number))
  9. applicant       (PK applicant_id;     UQ cnic, email)
 10. entry_test      (PK test_id)
 11. test_attempt    (PK (applicant_id, test_id);
                      FK applicant, entry_test)
 12. application     (PK application_id;   FK applicant, program, term;
                      UQ (applicant_id, program_id, term_id))
 13. offer           (PK offer_id;         FK application_id;
                      UQ application_id)
 14. student         (PK student_id;       FK program, applicant;
                      UQ applicant_id, email;
                      gpa DECIMAL(3,2) nullable)
 15. section         (PK section_id;       FK course, term, classroom, faculty;
                      UQ (course_code, term_id, section_label))
 16. enrollment      (PK (student_id, section_id);
                      FK student, section)

COLUMN CHEAT-SHEET (the most common query-time columns)
================================================================================
school         : school_id, school_name, abbreviation, established_year
faculty        : faculty_id, school_id, full_name, email,
                 designation ('Lecturer','Assistant Professor',
                              'Associate Professor','Professor')
program        : program_id, school_id, program_name,
                 degree_type ('BS','BE','BBA','BArch','MS','PhD'),
                 total_semesters, total_credits, total_seats
                 (Seed program codes: 'BSCS','BESE','BEE','BME','BIME',
                  'BBA','BSAF','BECE','BArch','BSAB','BChemE')

                 course_type ('Theory','Lab','Theory+Lab','Studio','Seminar'),
                 credit_hours (0..6), contact_hours (0..10)
prerequisite   : course_code, prereq_course_code
program_course : program_id, course_code, recommended_semester (1..14),
                 is_core (BOOLEAN)
term           : term_id, term_name ('Fall','Spring','Summer'),
                 academic_year, start_date, end_date
                 (Seed term codes: 'FA24','SP25','SU25','FA25','SP26','SU26',
                  'FA26','SP27','FA27','SP28')
classroom      : classroom_id, building, room_number, capacity
applicant      : applicant_id, full_name, cnic, email,
                 high_school_board ('FBISE','AKU-EB','Cambridge','Other'),
                 high_school_score (0..1100),
                 best_test_score (0..200, NULL allowed)
entry_test     : test_id,
                 test_type ('Engineering','CS','Business','Architecture',
                            'Biosciences','Chemical'),
                 test_date, total_marks
test_attempt   : applicant_id, test_id, score (>=0)
application    : application_id, applicant_id, program_id, term_id,
                 snapshot_hs_score, snapshot_best_test, aggregate_score,
                 submission_date,
                 status ('Pending','Selected','Waitlisted','Rejected',
                         'Enrolled','Declined')
offer          : offer_id, application_id, issue_date, expiry_date,
                 status ('Issued','Accepted','Declined','Expired')
student        : student_id, program_id, applicant_id, full_name, email,
                 current_semester (1..14), enrollment_date,
                 gpa (DECIMAL(3,2), 0.00..4.00, NULL until grades are posted)
section        : section_id, course_code, term_id, classroom_id, faculty_id,

enrollment     : student_id, section_id, attendance_percentage (0..100, NULL ok),
                 grade ('A','A-','B+','B','B-','C+','C','C-','D+','D','F' or NULL)

================================================================================
COMMON JOIN RECIPES
================================================================================

Student -> program / school (DIRECT, via denormalized student.program_id):
    student JOIN program ON program.program_id = student.program_id
            JOIN school  ON school.school_id   = program.school_id

Student -> applicant (name, CNIC, email — DIRECT, via student.applicant_id):
    student JOIN applicant ON applicant.applicant_id = student.applicant_id

Applicant's test scores:
    applicant JOIN test_attempt ON test_attempt.applicant_id = applicant.applicant_id
              JOIN entry_test   ON entry_test.test_id        = test_attempt.test_id

Top applicants by NET score:
    test_attempt JOIN applicant ON applicant.applicant_id = test_attempt.applicant_id
    ORDER BY test_attempt.score DESC

Applicant best NET score (per applicant):
    SELECT applicant_id, MAX(score) AS best_score
      FROM test_attempt GROUP BY applicant_id

Application -> program / school / applicant / term:
    application JOIN program   ON program.program_id     = application.program_id
                JOIN school    ON school.school_id       = program.school_id
                JOIN applicant ON applicant.applicant_id = application.applicant_id
                JOIN term      ON term.term_id           = application.term_id

Offer for a given application (and applicant):
    offer JOIN application ON application.application_id = offer.application_id
          JOIN applicant   ON applicant.applicant_id     = application.applicant_id

Student transcript (course + term + grade):
    enrollment JOIN section ON section.section_id = enrollment.section_id
               JOIN course  ON course.course_code = section.course_code
               JOIN term    ON term.term_id       = section.term_id
               JOIN student ON student.student_id = enrollment.student_id

Curriculum for a program:
    program_course JOIN course  ON course.course_code   = program_course.course_code
                   JOIN program ON program.program_id   = program_course.program_id

Prerequisites of a course (courses this course depends on):
    prerequisite JOIN course AS prereq
                   ON prereq.course_code = prerequisite.prereq_course_code
                 WHERE prerequisite.course_code = <code>

Section staffing (who teaches what, where, when):
    section JOIN course    ON course.course_code     = section.course_code
            JOIN term      ON term.term_id           = section.term_id
            JOIN faculty   ON faculty.faculty_id     = section.faculty_id
            JOIN classroom ON classroom.classroom_id = section.classroom_id

================================================================================
VIEWS, PROCEDURES, FUNCTIONS (defined in NUST.sql)
================================================================================
  VIEW student_transcript       - per-enrollment row with student name,
                                  program, course code/title, term, grade,
                                  attendance_percentage. Uses the direct
                                  student.program_id FK so no application
                                  join is required.

  VIEW classroom_utilization    - per-classroom: building, room_number,
                                  capacity, and the total number of
                                  sections hosted across every term.

  PROCEDURE accept_admission(p_student_id, p_program_id, p_applicant_id,
                             p_full_name, p_email, p_enrollment_date)
                                - transactionally inserts the student row.
                                  The AFTER INSERT trigger promotes the
                                  matching application from 'Selected' to
                                  'Enrolled' and flips the offer from 'Issued'
                                  to 'Accepted'. Any failure rolls back.

  PROCEDURE transfer_student_to_section(p_student_id, p_from_section,
                                        p_to_section)
                                - atomically moves a student from one section
                                  to another within a term, preserving
                                  attendance_percentage. If the destination
                                  section is at capacity, enforce_class_capacity
                                  raises SIGNAL and the EXIT HANDLER rolls back
                                  the DELETE so the student is not left without
                                  a seat.

  FUNCTION  is_eligible_for_engineering(p_applicant_id)
                                - Returns BOOLEAN. TRUE iff the applicant
                                  scored >= 140 on at least one Engineering-
                                  type entry_test (via test_attempt).
