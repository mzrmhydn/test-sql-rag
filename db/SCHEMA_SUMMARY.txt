================================================================================
NUST University Database - Schema Summary & Design Decisions
Scope: Undergraduate (Bachelor's) Programs (one 2-year MSIS entry for diversity)
DBMS: MySQL 8.0+ (InnoDB)
================================================================================

OVERVIEW
--------
The database models two modules that share the `Program` table:

  1. ACADEMIC MODULE
     School, Instructor, Program, Course, ProgramCourse,
     Student, Term, Classroom, Section, Enrollment

  2. ADMISSIONS MODULE
     Applicant, EntryTest, TestScore, Application

  3. FINANCE (shared)
     Fee  (unified ledger for Application fees and Student fees)

The bridge between modules: when an applicant's Application is accepted, a
new `Student` row is created whose ApplicationID FK points to that
Application. The Applicant and Program of a Student are therefore reached
through Student -> Application -> Applicant / Program (no denormalization).

KEY DESIGN DECISIONS
--------------------
1. All primary keys are AUTO_INCREMENT INTs. Human-readable codes live on
   secondary columns with UNIQUE constraints (Course.CourseCode,
   Program.DegreeType, School.Name, EntryTest.SeriesName, Term.TermName).
2. Student is 1:1 with an accepted Application; Student.ApplicationID is
   UNIQUE. Student does NOT carry ApplicantID or ProgramID directly.
3. Course is owned by a School (Course.SchoolID), NOT a Program.
   Program<->Course is a real M:N captured in ProgramCourse with
   CourseType ('Core'/'Elective') and Semester (1..10) attributes.
4. Unified Fee ledger. A single Fee table tracks both the per-application
   processing fee (ApplicationID set, StudentID NULL, FeeType='Application')
   and per-student tuition/hostel/library fees (StudentID set, ApplicationID
   NULL, FeeType IN ('Tuition','Hostel','Library')). An XOR CHECK enforces
   exactly one payer FK per row. A UNIQUE index on ApplicationID enforces
   "at most one Application fee per Application" (MySQL treats multiple NULLs
   as distinct in UNIQUE indexes, so student-fee rows with NULL ApplicationID
   are unaffected).
5. Enrollment separates Grade (nullable) from Status. Grade is NULL for
   'InProgress' and 'Withdrawn' rows and is a letter (A, A-, B+, B, B-, C+,
   C, C-, D+, D, F) for 'Completed' ones. A CHECK constraint enforces the
   Grade/Status pairing.
6. Applicant identity is stored in FirstName + LastName (no combined
   full-name column). HighSchoolMarks is a raw 0..1100 integer; there is
   no board-of-education column.
7. An Application's outcome lives in Application.Status, valid values:
   'Pending','Selected','Waitlisted','Rejected','Enrolled','Declined'.
   The trigger AutoUpdateApplicationStatus promotes 'Selected' to
   'Enrolled' whenever a Student row referencing that Application is
   inserted.
8. Classroom capacity is enforced by the BEFORE INSERT trigger
   EnforceClassCapacity: any Enrollment that would push a Section past
   its hosting Classroom's Capacity is rejected with SQLSTATE 45000.
9. ON DELETE CASCADE is used for the junction tables (TestScore,
   Application, ProgramCourse) so deleting a parent sweeps its junction
   rows. Other FKs use the MySQL default (RESTRICT).
10. Indexes exist on high-traffic FK/filter columns (Enrollment.StudentID,
    Section.CourseID/TermID/InstructorID, Application.Status/ProgramID,
    TestScore.TestID, Course.SchoolID, ProgramCourse.CourseID,
    Fee.StudentID, Fee.FeeType).
11. The schema ships with two views (StudentTranscript, ClassroomUtilization),
    two stored procedures (GenerateTuitionChallan, AcceptAdmission), and one
    function (IsEligibleForEngineering).

TRIGGERS
--------
- EnforceClassCapacity           BEFORE INSERT on Enrollment
    (rejects the row if the hosting Section's Classroom is at capacity)
- AutoUpdateApplicationStatus    AFTER  INSERT on Student
    (flips the referenced Application from 'Selected' to 'Enrolled')

GRADE LETTERS
-------------
Allowed values on Enrollment.Grade:
    A, A-, B+, B, B-, C+, C, C-, D+, D, F
Grade NULL means the row is 'InProgress' or 'Withdrawn'.
Note: There is no CGPA calculation helper; Student.CGPA is stored directly
as a DECIMAL(4,2) in [0.00, 4.00] and is maintained externally.

================================================================================
EXACT CARDINALITIES BETWEEN ENTITIES
================================================================================

ACADEMIC MODULE
---------------
School       1 ---- 1..*  Program         (Program.SchoolID NOT NULL)
School       1 ---- 0..*  Instructor      (Instructor.SchoolID NOT NULL)
School       1 ---- 0..*  Course          (Course.SchoolID NOT NULL)
School       1 ---- 0..*  Classroom       (Classroom.SchoolID NOT NULL)

Program      *  ---- *    Course          (via ProgramCourse; junction adds
                                           CourseType + Semester)

Course       1 ---- 0..*  Section         (Section.CourseID NOT NULL)
Term         1 ---- 0..*  Section         (Section.TermID NOT NULL)
Instructor   1 ---- 0..*  Section         (Section.InstructorID NOT NULL)
Classroom    1 ---- 0..*  Section         (Section.ClassroomID NOT NULL)

Student      *  ---- *    Section         (via Enrollment; junction carries
                                           Grade + Status)

ADMISSIONS MODULE
-----------------
Applicant    1 ---- 0..*  Application     (Application.ApplicantID NOT NULL)
Program      1 ---- 0..*  Application     (Application.ProgramID NOT NULL)
Applicant    *  ---- *    EntryTest       (via TestScore; junction carries
                                           Score 0..200)

CROSS-MODULE LINK
-----------------
Application  1 ---- 0..1  Student         (Student.ApplicationID UNIQUE and
                                           NOT NULL; a Student always exists
                                           for exactly one Application)

FEE (shared)
------------
Application  1 ---- 0..1  Fee             (Application-type fees; UNIQUE on
                                           Fee.ApplicationID)
Student      1 ---- 0..*  Fee             (Tuition/Hostel/Library fees; many
                                           rows per Student)

================================================================================
TABLE INVENTORY
================================================================================
  1. School         (PK SchoolID;        UQ Name)
  2. Program        (PK ProgramID;       FK SchoolID; UQ (SchoolID, ProgramName))
  3. Applicant      (PK ApplicantID;     UQ Email)
  4. EntryTest      (PK TestID;          UQ SeriesName)
  5. TestScore      (PK TestScoreID;     UQ (ApplicantID, TestID); FK Applicant, EntryTest)
  6. Application    (PK ApplicationID;   UQ (ApplicantID, ProgramID); FK Applicant, Program)
  7. Student        (PK StudentID;       UQ ApplicationID; FK Application)
  8. Fee            (PK FeeID;           UQ ApplicationID; FK Application, Student; XOR payer)
  9. Instructor     (PK InstructorID;    UQ Email; FK School)
 10. Course         (PK CourseID;        UQ CourseCode; FK School)
 11. ProgramCourse  (PK (ProgramID, CourseID); FK Program, Course)
 12. Term           (PK TermID;          UQ TermName)
 13. Classroom      (PK ClassroomID;     UQ (SchoolID, RoomNumber); FK School)
 14. Section        (PK SectionID;       UQ (CourseID, TermID, SectionName); FK Course, Term, Instructor, Classroom)
 15. Enrollment     (PK EnrollmentID;    UQ (StudentID, SectionID); FK Student, Section)

================================================================================
COLUMN CHEAT-SHEET (the most common query-time columns)
================================================================================
School        : SchoolID, Name, Location, EstablishedYear
Program       : ProgramID, SchoolID, ProgramName, DegreeType, DurationYears, TotalSeats
Applicant     : ApplicantID, FirstName, LastName, Email, Phone, DOB,
                HighSchoolMarks, City
EntryTest     : TestID, SeriesName, TestDate,
                TestType ('Engineering','Business','Architecture',
                          'Biosciences','Chemical')
TestScore     : TestScoreID, ApplicantID, TestID, Score
Application   : ApplicationID, ApplicantID, ProgramID, ApplicationDate,
                Preference (1..5),
                Status ('Pending','Selected','Waitlisted','Rejected',
                        'Enrolled','Declined')
Student       : StudentID, ApplicationID, EnrollmentDate, CGPA,
                Status ('Active','Graduated','Suspended','Withdrawn')
Fee           : FeeID, ApplicationID, StudentID,
                FeeType ('Application','Tuition','Hostel','Library'),
                Amount, PaymentDate,
                Method ('Bank','Online','Cheque','Cash')
Instructor    : InstructorID, SchoolID, FirstName, LastName,
                Title ('Lecturer','Assistant Professor',
                       'Associate Professor','Professor'), Email, HireDate
Course        : CourseID, SchoolID, CourseCode, CourseName, Credits
ProgramCourse : ProgramID, CourseID,
                CourseType ('Core','Elective'), Semester (1..10)
Term          : TermID, TermName (e.g. 'Fall 2025'), StartDate, EndDate
Classroom     : ClassroomID, SchoolID, RoomNumber, Capacity,
                RoomType ('Lecture','Lab','Studio','Hall')
Section       : SectionID, CourseID, TermID, InstructorID, ClassroomID,
                SectionName
Enrollment    : EnrollmentID, StudentID, SectionID, Grade,
                Status ('InProgress','Completed','Withdrawn')

================================================================================
COMMON JOIN RECIPES
================================================================================

Student -> Applicant (name, contact):
    Student JOIN Application ON Application.ApplicationID = Student.ApplicationID
            JOIN Applicant   ON Applicant.ApplicantID   = Application.ApplicantID

Student -> Program / School:
    Student JOIN Application ON Application.ApplicationID = Student.ApplicationID
            JOIN Program     ON Program.ProgramID         = Application.ProgramID
            JOIN School      ON School.SchoolID           = Program.SchoolID

Applicant -> best NET score:
    Applicant LEFT JOIN TestScore ON TestScore.ApplicantID = Applicant.ApplicantID
    (use MAX(TestScore.Score) in a GROUP BY)

Student transcript (course + term + grade):
    Enrollment JOIN Section ON Section.SectionID = Enrollment.SectionID
               JOIN Course  ON Course.CourseID   = Section.CourseID
               JOIN Term    ON Term.TermID       = Section.TermID

Curriculum for a program:
    ProgramCourse JOIN Course  ON Course.CourseID   = ProgramCourse.CourseID
                  JOIN Program ON Program.ProgramID = ProgramCourse.ProgramID

Application-type fees:
    Fee WHERE FeeType = 'Application'      (key: Fee.ApplicationID)
Student-type fees:
    Fee WHERE FeeType IN ('Tuition','Hostel','Library')
        (key: Fee.StudentID; Fee.ApplicationID IS NULL)

================================================================================
VIEWS, PROCEDURES, FUNCTIONS (defined in NUST.sql)
================================================================================
  VIEW StudentTranscript       - per-enrollment row with student name, program,
                                 course, term, grade, and enrollment status
  VIEW ClassroomUtilization    - sections hosted per classroom with capacity

  PROCEDURE GenerateTuitionChallan(p_student_id, p_amount)
                                - inserts a Tuition Fee row for the student
  PROCEDURE AcceptAdmission(p_application_id, p_enrollment_date, p_tuition_amount)
                                - transactionally creates Student + first
                                  Tuition Fee; the Application is auto-
                                  promoted to 'Enrolled' by the trigger

  FUNCTION  IsEligibleForEngineering(p_applicant_id)
                                - TRUE iff applicant's best Engineering-type
                                  NET score is >= 140
